# 베이커리 PoC — 보충 데이터 요청

> 광교 매장(점포코드 1000000047) 5년치 데이터로 PoC 진행 결과, PM 기획서의 일부 평가 지표(MASE, Coverage, Waste rate, 발주 적중률 등)를 직접 측정하려면 추가 데이터가 필요합니다.
>
> 또한 데이터 재분석으로 "품절 → 매출 손실" 가정이 광교에서 검증되지 않아 사업 가치 추정 framework 자체를 재정의 중 (`docs/poc_framework_pivot.md` 참조). 진짜 PoC 가치 검증에 보충 데이터가 결정적.

---

## TL;DR — 최우선 3개

1. **폐기 수량** (매장×일×품목 단위) — PM 평가 지표 Waste rate KPI 직접 측정 + production model 진짜 ROI 검증
2. **온·오프라인 채널 구분** (배민 B마켓 등) — 시간 패턴·substitution 분석의 noise 제거
3. **사전 예약 정보** (cake/케이크 등) — 시즌 특수 제품 모델링 (새 framework 3대 risk 중 하나)

위 3개 없으면 사업 KPI 정확 검증 불가. 4~11번은 받을수록 좋음.

---

## 1순위 — 폐기 수량 (또는 유추용 sub 데이터)

### 옵션 A — 폐기 수량 직접 받기 (최선)

**최소 schema**:

| 컬럼 | 타입 | 예시 |
|---|---|---|
| 점포코드 | string | 1000000047 |
| 판매일자 | date | 2024-01-15 |
| 품목코드 | string | 151100000247 |
| **폐기수량** | int | 5 |

**가능하면 추가** (분석 풍부도 ↑):

| 컬럼 | 의미 | 사용처 |
|---|---|---|
| 폐기시점 | 당일 마감 / 다음날 / 일주일 후 | 당일 폐기 vs 재고 이월 분리 |
| 폐기사유 | 유통기한 / 손상 / 기타 | 진짜 unavoidable waste 식별 |
| 회수구분 | 마감세일 / 직원식사 / 기부 / 일반폐기 | 회수율 추정 → 실 cost 정확화 |
| 회수금액 | 할인판매·기부세액공제 등 회수 가치 | net waste cost 계산 |

### 옵션 B — 폐기량 유추용 sub 데이터 (옵션 A 어려운 경우)

폐기량은 회계 항등식으로 자동 계산 가능:

```
시작재고 + 입고 = 판매 + 폐기 + 종료재고
→ 폐기 = 입고 + 시작재고 − 종료재고 − 판매(이미 보유)
```

**옵션 B-1 — 일별 입고량만 받기** (가장 가벼움, 광교에 최적)

광교 품목정보에 `당일폐기여부=Y`가 397/417 (95%). 당일폐기 품목은 다음날 재고 이월 0이므로:

```
당일폐기 품목:   폐기 = 입고 − 판매
```

→ **일별 입고량 하나만 받아도 광교 95% 품목 폐기량 정확 계산**.

필요 schema:

| 컬럼 | 의미 |
|---|---|
| 점포코드 | |
| 입고일자 | date |
| 품목코드 | |
| **입고수량** | 그날 매장에 들어온 수량 (제조량 or 발주량) |

발주·제조 시스템에 이미 있는 데이터일 가능성 큼 (매장 운영자가 매일 만드는 발주서).

**옵션 B-2 — 입고 + 시작·종료 재고**

당일폐기 N 품목까지 정확히 계산하려면 재고 이월 분리:

| 추가 컬럼 | |
|---|---|
| 시작재고 | 그날 영업 시작 시점 |
| 종료재고 | 그날 영업 종료 시점 (= 다음날 시작재고) |

**옵션 B-3 — 정기 정산 손실 보고**

매장이 정기 회계 정산 시 만드는 일별 손실/폐기 합계 보고서. 회수 구분은 못 잡지만 총 폐기량은 추출 가능. POS 외부 시스템에 보통 존재.

### 기간

2024-01-01 ~ 2025-12-31 (학습 윈도우 align). 가능하면 2021년부터 5년 전체.

### 왜 1순위인가

- PM 기획서 평가 지표 (3)의 첫 KPI "Waste rate 일 20개→8개"를 **개수 단위로 직접 측정 가능**
- 현재 PoC는 `waste_cost = max(yhat − sold, 0) × unit_price × 0.30` 추정 중 → 0.30 (원가율)이 광교 실제와 다를 수 있음
- 새 framework 발견: v2_q85 production model의 진짜 가치는 **매출 회복이 아니라 폐기 절감**. 검증 source 없으면 가치 추정 불가
- 회수율 모르면 실 손실의 over/under-estimate 가능

### 우선순위 (옵션 간)

| 옵션 | 필요 데이터 | 광교 정확도 | 매장 부담 |
|---|---|---|---|
| **A** (직접) | 폐기수량 | ★★★ 최고 | 중간 |
| **B-1** (입고만) | 일별 입고량 | **★★★ 최고 (95% 품목)** | **낮음 (발주 시스템에 있음)** |
| B-2 (입고+재고) | 입고 + 시작/종료재고 | ★★★ 최고 (전체) | 높음 |
| B-3 (정산보고) | 일별 손실 보고 | ★★ 중간 | 낮음 |

→ **A 또는 B-1 중 가용한 쪽으로 진행. 광교는 당일폐기 비중이 95%라 B-1만으로도 충분히 정확**.

## 2순위 — 온·오프라인 채널 구분

### 필요 형식

판매정보 시트의 각 row에 channel 컬럼 추가:

| 컬럼 | 값 |
|---|---|
| channel | `store` (매장 픽업 in-store) / `delivery_baemin` / `delivery_coupang` / `pickup_app` / `other` |

또는 별도 매핑 테이블: `(점포, 영수증번호, 채널)`

### 왜 필요한가

- 배민 B마켓 등 외부 채널 매출은 시간 분포 패턴이 매장 직접 판매와 다름
- 영수증당 평균 1.86 품목 / hour profile / substitution 분석이 채널 혼합으로 왜곡됨
- DiD substitution 분석에서 채널별 noise 식별 안 됨

## 3순위 — 사전 예약 정보 (cake/시즌 특수 제품)

### 필요 schema

| 컬럼 | 의미 |
|---|---|
| 점포코드 | |
| 예약일 | 손님이 주문한 날 |
| 인도일 | 받기로 한 날 |
| 품목코드 | |
| 수량 | |
| 예약금액 | |
| 상태 | 정상수령 / 취소 / no-show |

### 왜 필요한가

- 새 framework 핵심: 베이커리는 카테고리 단위 substitute지만 **케이크는 진짜 substitute 어려움** (생일·발렌타인·크리스마스 목적)
- 예약 매출 비중 모르면 일반 판매와 분리 불가 → cake 시즌 특수 모델 구축 불가
- 예약 미수령·취소율도 폐기 추정에 직접 영향

## 4순위 — 영업시간 + 휴무일

### 필요 schema

매장별 정상 영업시간 (점포정보 시트 확장):

| 컬럼 | |
|---|---|
| 점포코드 | |
| 평일 영업시작 / 영업종료 | hh:mm |
| 주말 영업시작 / 영업종료 | hh:mm (다른 경우) |

임시 휴무일 (별도 시트):

| 컬럼 | |
|---|---|
| 점포코드 | |
| 휴무일자 | date |
| 사유 | 정기휴무 / 명절 / 자체휴무 / 긴급휴무 |

### 왜 필요한가

- 현재 PoC: `open_hour=7, close_hour=22` hardcode 사용
- 일부 일자 sold=0인 게 정말 휴무인지 매출 0인지 구분 불가 → potential_demand 보정 정확도에 영향
- 매장별 영업시간 다르면 다매장 확장 시 hardcode 무의미

## 5순위 — 실시간 재고 / 일별 초기 재고

### 필요 schema (옵션 A: 일별 시작 재고만 — 최소)

| 컬럼 | |
|---|---|
| 점포코드 | |
| 판매일자 | |
| 품목코드 | |
| **시작재고수량** | 그 날 영업 시작 시점 재고 |

### 옵션 B: 시각별 재고 (더 좋음)

| 컬럼 | |
|---|---|
| 점포코드 | |
| 판매일자 | |
| 시각 (hh:mm) | |
| 품목코드 | |
| 재고수량 | |

### 왜 필요한가

- `capacity` 현재 추정값(sold_units max). 진짜 capacity 식별 시 자기실현적 사이클 천장 정확화
- 폐기 수량 + 시작재고 + 판매수량 = `재고 검증` 데이터 일관성 확보
- 폐기 데이터(1순위)가 있으면 `sold + waste = production`으로 일부 우회 가능 → 5순위로 강등

## 6순위 — 공식 카테고리 분류

### 필요 schema

품목정보 시트에 컬럼 추가:

| 컬럼 | 예시 값 |
|---|---|
| **공식 카테고리** | 식빵류 / 페이스트리 / 케이크 / 샌드위치 / 디저트 / 음료 / 시즌특수 |
| 하위 분류 (있다면) | 단팥빵 / 크림빵 / 식사빵 / 시즌 한정 등 |

### 왜 필요한가

- 현재 우리가 POS메뉴명 추론으로 5개 카테고리(bread/pastry/cake/sandwich/sweets) 부여 중
- 매장 공식 분류와 다르면 "카테고리 = 한 묶음" framework 기반 무너짐
- 운영자가 보는 카테고리 = 의사결정 단위 = 진짜 사업 KPI 측정 단위

## 7순위 — 다른 매장 데이터

### 필요 schema

광교와 동일 (판매정보 + 품목정보 + 점포정보 + 품절정보 + 위 1~6번 보충 데이터).

### 매장 선택 가이드 (지역·규모 분산)

| 매장 유형 | 효과 |
|---|---|
| 서울 핵심 상권 (강남·홍대) | v3 외부 features (생활인구·소비) 동작 확인 |
| 오피스 상권 (여의도·광화문) | sandwich 카테고리 시간 패턴 분석 |
| 주택가 매장 | bread 안정성 + 가족 구매층 |
| 신규 매장 (1~2년) | cold start 시나리오 검증 |

### 왜 필요한가

- 광교 1매장으로는 v3 외부 features 모두 `store_id`와 collinear → 학습 정보 없음
- DiD substitution도 단일 매장 power 부족
- 다매장 cross-validation으로 광교의 본질 vs 우연 구분 가능

## 8~11순위 — 받을수록 좋음

| 순위 | 데이터 | 짧은 이유 |
|---|---|---|
| 8 | **매장 규모** (영업면적, 좌석수, 직원수, 가맹/직영) | 다매장 baseline multiplier 학습 |
| 9 | **유통기한 정확화** (품목별 일수) | 폐기 비용 정확화 (현재 당일폐기 Y/N만) |
| 10 | **제조·납품 구분** (자체제조 / 외부납품) | 수급 lead time 모델링 |
| 11 | **프로모션 매장·시간대 적용** | 할인 효과 noise 제거 |

## 받을 형식 통합 제안

가능하면 **하나의 xlsx** 또는 **parquet/csv 셋**으로:

```
보나비_보충데이터_<날짜>.xlsx
├─ 폐기정보      (1순위)
├─ 채널정보      (2순위)
├─ 예약정보      (3순위)
├─ 영업시간      (4순위)
├─ 휴무일자      (4순위)
├─ 재고정보      (5순위, 가능 시)
├─ 카테고리매핑  (6순위)
└─ [다른매장]    (7순위, 별도 파일도 OK)
```

매장 신상정보 (점포정보 확장)는 점포정보 시트에 컬럼 추가하는 형태가 깔끔.

## 우리 측 작업 (데이터 도착 시 즉시 진행)

| 도착 데이터 | 즉시 작업 |
|---|---|
| 1. 폐기 수량 | `business_metrics.py` 추정 → 실측 교체, Waste rate KPI 측정 코드 추가, v2_q85 ROI 재계산 |
| 2. 채널 | 채널별 hour profile + DiD 재추정 |
| 3. 예약 | cake 시즌 특수 모델 분리, 예약/일반 hour profile 분리 |
| 4. 영업시간/휴무 | open/close 동적, sold=0 일자 진위 분리 |
| 5. 재고 | capacity 정확화, 자기실현적 사이클 재검증 |
| 6. 카테고리 | framework_pivot.md 정렬, inter-category substitution 재검증 |
| 7. 다른 매장 | v3 외부 features 가치 검증, DiD power 회복, 다매장 multiplier 학습 |

위 작업은 모두 코드·인프라가 준비되어 있어 데이터만 들어오면 즉시 적용 가능합니다.
