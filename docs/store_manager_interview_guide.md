# 광교 점장 인터뷰 가이드 (진행자용)

> **목적**: 우리가 5년 데이터로 발견한 것들을 점장이 **확인·설명**하게 하고(가정 검증), 현재 **발주/생산을 어떻게 결정하는지**(전향 실측 비교의 baseline) 파악하고, 모델이 막힌 **데이터 갭**(원가율·카테고리·예약)을 확보한다.
>
> **대상**: 광교점 점장 (우리 PoC 검증 매장). **시간**: 60~90분 권장.
> **원칙**: 우리 분석 결과를 먼저 말하지 말 것 — 점장이 스스로 설명하게 유도(리딩 질문 금지). 우리 가설은 답을 들은 뒤 대조.

---

## 0. 진행자 치트시트 — 우리가 데이터로 발견한 것 (검증 대상)

인터뷰 중 점장 답변과 대조할 우리 발견. **점장에게 먼저 읽어주지 말 것.** ★ = 최우선.

| # | 데이터 발견 (광교 5년) | 상태 | 인터뷰에서 확인할 것 |
|---|---|---|---|
| F1 ★ | **거의 매일 폐기 발생**(폐기>0인 날 97%). 카테고리가 통째 동나는 날 거의 없음(빵/페이스트리 완판 2.68%, 조기소진 ≈0%) | 우리 추론: 과잉생산 구조 | 의도적 과잉생산인가? 왜? |
| F2 ★ | **저녁 마감할인이 5년 상시**. 저녁 판매의 91~93%가 할인품. 할인해도 손님 수(footfall)는 안 줄어듦. **2025-01-17에 할인율 30%→20% 변경** | α(진짜수요비율) 데이터로 미확정 | 손님이 할인 때문에 오나, 원래 살 손님인가 |
| F3 ★ | 현행 발주=**sold-out 지수 기반 증산/감산**(과거 3주, 요일별) — 실무자 미팅서 확인. 시스템에 예측량/발주량 입력 가능 | baseline 파악됨(로직 전달 예정) | 점장은 이 시스템을 **어떻게 쓰나·수동 개입** 확인 |
| F4 | 크리스마스 판매가 5년 ~330개로 안정(+23%). **주말 xmas에 활성 빵 SKU가 44→37개로 감소** | 우리 가설: 케이크 인력쏠림→빵 공급상한 | xmas에 케이크 때문에 빵 생산 제약? |
| F5 | 카테고리 비중: **페이스트리 68.5% / 빵 27.4% / 샌드위치 4%**. 세 카테고리가 요일·이벤트 무관하게 거의 같은 비율로 움직임 | 합산 예측 근거 | 손님이 카테고리 내 대체 구매? |
| F6 | 예약/대량주문이 **POS에 기록 안 됨**(추정치로만 검출, 5년 ~1,615개). 케이크 예약 할인코드는 광교 사용 0건 | 데이터 갭 | 예약·케이터링 어떻게 처리·기록? |
| F7 | "냉동생지_24" 메뉴는 실제 판매 완제품(2024 신메뉴, 냉동반죽 구움). 일부 "(생지)" 품목은 거래 0건 유령 | 확인 필요 | 냉동생지 = 손님에 파는 완제품? |
| F8 | 원가율/마진 데이터 없음. 발주 패턴 역산 시 원가율 ~0.20(빵)/0.28(페이스트리)처럼 행동(대략 합리적) | 데이터 갭 | 원가율은 점장이 아나, 본사인가 |
| F9 | 품목의 90%가 안정(Smooth) 수요, 럼피/불안정 거의 없음. 단종·시즌 품목 다수 | — | 메뉴 교체 주기·결정 주체 |

## 0.5 실무자(파트장) 미팅 반영 (2026-07-14) — 이 인터뷰의 조정점

킥오프 실무자 미팅에서 이미 확보한 지식. **점장 인터뷰는 이걸 현장에서 확인·심화하는 데 집중**한다. (원문: 아띠제 PoC 킥오프 실무자 미팅 정리 페이지)

**이미 확인됨 → 점장에겐 "현장에서 어떻게 쓰나"만 확인**
- **현행 발주 시스템 = sold-out 지수 기반 증산/감산**(과거 3주, 요일별 등 다인자). 로직 전달 예정. → 점장에겐 "이 제시량을 그대로 쓰나, 조정하나, 언제 개입하나"만.
- **시스템에 예측량/발주량 둘 다 입력 가능**(예측량 넣으면 발주량 자동 산정) — 우리 산출물 형태 결정용, 점장 질문 아님.
- **생산 배수 제약**: 냉동생지 1개=N개 분량 → 3/6/9 배수로만 생산. 상품별 룰 존재(데이터 요청 대상).
- **예약(bulk) 검출 아이디어**: ①하루 30%↑ 튀면 평균대체 ②판매+폐기>발주=추가발주(bulk). → 현장 검증 대상.

**점장에게 새로 꼭 물을 것 (실무자 미팅으로 드러남)**
- **지역 이벤트 ★★**: **수원 월드컵경기장 행사(특히 싸이 흠뻑쇼)** 때 광교 애비뉴프랑에 인파 몰려 매출 급증. → 어떤 이벤트가, 얼마나, 어떻게 아는지 (테마 F에 반영).
- **배수 생산 현실**: 어떤 빵이 3/6/9 배수 생산인지, 그래서 원하는 수량을 못 맞추는지 (테마 A).
- **신제품 배치 정책**: 신제품은 덜 팔려도(폐기 감수) 일단 많이 배치 — 맞는지, 언제까지 그렇게 (테마 E).
- **예약이 데이터에 어떻게 남나**: 판매·발주·폐기 중 어디에? 실무자 아이디어(30%↑, 판매+폐기>발주) 현장 확인 (테마 G).
- **수동 개입**: "다음 주 이슈 예상되면 덜 발주 문의" — 얼마나 자주, 어떤 근거로 (테마 A).

**원칙(실무자 확인)**: 이벤트는 **아주 명확한 것만** 반영 — 세부까지 다 넣으면 점장 운영 부담 과다. 자동화(웹 수집→승인→피처)는 장기 목표.

## 1. 오프닝 (5분)

- 감사 인사 + 목적 프레이밍: "매장 판매/발주 데이터를 분석 중인데, **숫자만으로는 알 수 없는 실제 운영 방식**을 여쭤보러 왔습니다. 정답이 있는 게 아니라 실제로 어떻게 하시는지가 궁금합니다."
- **녹음/기록 동의** 확인.
- 워밍업: "이 매장에서 얼마나 근무하셨나요? 하루 흐름이 대략 어떻게 되나요?" (라포 + 근속으로 답변 신뢰도 가늠)

## 2. 테마 A — 발주/생산 프로세스 ★최우선 (15분)

> **왜**: 4주 전향 실측에서 "기존 방식 vs 우리 모델"을 비교하는데, **기존 방식이 뭔지**를 정확히 알아야 baseline이 선다. 지금 이게 데이터에 전혀 없음(F3).

- **Q A1 ★**: "내일(또는 오늘) 각 빵을 **몇 개 만들지 어떻게 정하세요?**"
  - ↳ 갈림길: 순수 감/경험 · 최근 판매 평균(며칠치?) · POS/본사 시스템 추천값 · 정해진 레시피 수량
  - ↳ 후속: "시스템이 숫자를 주나요, 점장님이 직접 정하나요? 시스템 값을 얼마나 믿고 조정하세요?"
- **Q A2**: "그 결정을 **언제, 누가** 하나요?" (전날 저녁 / 당일 새벽 / 제빵사와 상의)
- **Q A3**: "**요일·날씨·행사**에 따라 수량을 바꾸시나요? 어떻게요?"
  - ↳ 대조: F4(특수일), 날씨 반응(우리 데이터상 약함)
- **Q A4 ★**: "만들 수 있는 양에 **한계**가 있나요? (오븐·인력·시간) 다 팔릴 것 같아도 그만큼 못 만드는 경우가 있나요?"
  - ↳ 대조: F4 크리스마스 공급상한 가설. 이게 "수요를 못 맞춘 게 아니라 못 만든 것"인지 판별.
- **Q A5**: "**냉동생지/구움과자**는 어떻게 운영하세요? (냉동 반죽을 당일 굽는지, 몇 시에)"
  - ↳ 대조: F7. "냉동생지_24" 신메뉴 정체 확인.
- **Q A6 ★**: "**정해진 묶음 단위로만 만들 수 있는 빵**이 있나요? (예: 냉동생지 1개가 3개 분량이라 3·6·9개씩만 생산) 어떤 빵이 그런가요?"
  - ↳ 대조: 실무자 확인(배수 생산). 우리 발주 반올림에 품목별 배수 제약 반영 필요 — 어떤 품목이 대상인지.
  - ↳ 후속: "그래서 딱 필요한 만큼 못 만들고 남거나 모자란 경우가 있나요?"
- **Q A7**: "발주 시스템이 제시하는 수량을 **그대로 쓰세요, 조정하세요?** '다음 주에 이슈 있겠다' 싶으면 **덜 발주 요청**한다고 들었는데, 얼마나 자주 그러시나요? 어떤 근거로요?"
  - ↳ 대조: F3. 현행 시스템(sold-out 지수) 위에 점장 수동 개입 정도 = baseline 정밀화.
- **Q A8**: "제품마다 **진열되는 시간이 다른가요?** (개장하자마자 다 있는 게 아니라 순차적으로)"
  - ↳ 대조: 실무자 확인(제품별 진열 시간). 시간대 판매·품절 해석에 반영.

## 3. 테마 B — 과잉생산 & 폐기 ★최우선 (12분)

> **왜**: 데이터상 거의 매일 폐기가 남음(F1). 우리 모델의 핵심 가치가 "폐기 절감"이라, **과잉생산이 의도인지 실수인지**가 결정적. 의도라면 왜(진열·매진회피)인지가 α·발주정책과 직결.

- **Q B1 ★**: "하루 마감하면 보통 **빵이 좀 남나요, 거의 다 나가나요?**"
  - ↳ 그냥 열어둠. 점장이 스스로 "남는다"고 하면 → B2로.
- **Q B2 ★**: "남는 걸 알면서도 **넉넉히 만드는** 편인가요? 그렇다면 왜요?"
  - ↳ 갈림길: ① 진열이 비면 안 돼서(풍성함) ② 품절나서 손님 놓치는 게 더 싫어서 ③ 본사 지침 ④ 예측이 어려워서 안전하게
  - ↳ 대조: F1. ②라면 "매진 회피 선호"가 강함 → 우리 발주 분위수(q) 정책 정당화.
- **Q B3**: "**폐기가 얼마나 나오면** '많다'고 느끼세요? 목표치나 관리 기준이 있나요?"
- **Q B4 ★**: "남은 빵은 어떻게 처리하나요? (마감할인 → 그래도 남으면 폐기?)" → 테마 C로 자연 연결.
- **Q B5**: "폐기가 **점장 평가나 원가**에 잡히나요? 신경 쓰는 지표인가요?"
  - ↳ 대조: F8 원가율. 폐기 비용 인식 수준.

## 4. 테마 C — 마감할인 (α) ★최우선 (12분)

> **왜**: "마감할인으로 팔린 빵 중 **원래 정가에도 살 손님 비율(α)**"을 데이터로 못 가림(F2). 이게 실수요 추정의 핵심 가정. 점장 직관이 유일한 추가 증거.

- **Q C1**: "**마감할인은 어떻게** 하세요? 몇 시부터, 몇 %, 누가 정하나요?"
  - ↳ 대조: F2. 우리는 저녁 상시·2025년 20%(이전 30%)로 봄 — 확인.
- **Q C2 ★**: "저녁에 **할인 빵을 사는 손님들** — 원래 그 시간에 와서 **정가라도 살 손님**일까요, 아니면 **할인이라서** 사는 손님일까요? 점장님 느낌엔?"
  - ↳ 갈림길: "원래 살 손님이 대부분"=높은 α / "할인 노리고 옴"=낮은 α. **직접 리딩 금지** — 열린 질문으로.
- **Q C3**: "할인 빵 사는 김에 **정가 다른 것도** 같이 사가나요? (음료·다른 빵)"
  - ↳ 대조: 우리 basket 분석이 저녁 재고 부족으로 막힘. 점장 관찰로 보완.
- **Q C4**: "2025년에 할인율을 **30%→20%로 낮췄을 때** 저녁 매출/손님이 줄던가요?"
  - ↳ 대조: F2 depth-invariant 발견. 점장 체감과 대조.

## 5. 테마 D — 품절 경험 & 대체 ★ (12분)

> **왜**: 데이터상 카테고리가 통째 동나는 일은 드묾(F1). 하지만 **손님이 품절 시 대체 구매하는지**(카테고리=한 묶음 수요, F5)와 **하루 중 리필**(재생산) 여부는 데이터로 애매 — 점장 확인 필요.

- **Q D1**: "빵이 **중간에 떨어지는** 경우가 자주 있나요? 어떤 빵이, 보통 몇 시쯤?"
  - ↳ 대조: F1. 우리는 거의 안 동난다고 봄 — 점장 체감 대조.
- **Q D2 ★**: "인기 빵이 떨어지면 손님이 **다른 빵으로 바꿔** 사나요, 아니면 **그냥 가나요?**"
  - ↳ 대조: F5 카테고리 내 대체(수요 흡수). "비슷한 빵으로 바꾼다"=흡수 지지.
- **Q D3 ★**: "빵이 떨어지면 **그 자리에서 더 굽나요(리필)?** 하루에 여러 번 만드는 품목이 있나요?"
  - ↳ 대조: 우리 품절 데이터의 "하루 다중 품절이벤트"(리필→재품절) 원인 확인. 매우 중요.
- **Q D4**: "품절나서 손님 놓치는 게 **얼마나 자주/심각**한가요? 폐기랑 비교하면 뭐가 더 신경 쓰이세요?"
  - ↳ 대조: 매진 cost vs 폐기 cost 상대 가중치(우리 발주정책 w).

## 6. 테마 E — 카테고리·품목·메뉴 (10분)

> **왜**: 공식 카테고리 미제공(우리는 빵/페이스트리/샌드위치로 추론, F5). 냉동생지·단종·잼/케이크 채널 등 데이터 경계 확인.

- **Q E1**: "매장 빵을 크게 **몇 가지 종류로** 나눠서 관리하세요? (식사빵/단과자/샌드위치 …) 공식 분류가 있나요?"
  - ↳ 대조: F5. 우리 3분류가 맞는지.
- **Q E2**: "**메뉴가 얼마나 자주** 바뀌나요? 신제품 넣고 빼는 건 누가 정하나요(점장/본사)?"
  - ↳ 대조: F9 단종·시즌 다수.
- **Q E2b ★**: "**신제품**은 처음에 얼마나 만드세요? 아직 반응을 모르는데 **일부러 넉넉히** 놓고 좀 남아도(폐기) 감수하나요? 언제까지 그렇게 하나요?"
  - ↳ 대조: 실무자 확인(신제품=더 배치, 반응 탐색 위해 폐기 감수). 우리 모델이 신제품은 폐기 페널티를 낮춰야 하는지 결정. 앞으로 신제품 많이 출시 예정이라 중요.

## 7. 테마 F — 특수일 (10분)

> **왜**: 크리스마스·추석 등에 판매가 튐(F4). 우리는 "이벤트 고유 레벨"로 모델 보정을 넣었는데, **원인(수요 급증 vs 케이크 인력쏠림 공급상한)**을 점장이 확정해줘야 함.

- **Q F1 ★**: "**크리스마스** 때 매장이 어떻게 달라지나요? 케이크 주문이 몰리면 **빵 만들 인력·오븐이 부족**해지나요?"

- **Q F1b ★★**: "**매장 주변 행사** 때문에 매출이 확 오르는 날이 있나요? (수원 월드컵경기장 행사·**싸이 흠뻑쇼**·광교 애비뉴프랑 자체 행사 등)"
  - ↳ 대조: 실무자 확인(싸이 흠뻑쇼 날 급증). **광교 특화 지역 이벤트**가 명절보다 클 수 있음. 어떤 이벤트가·얼마나·미리 아는 경로(공연 일정 등).
  - ↳ 후속: "그런 날을 **미리 알고 준비**하시나요? 어디서 정보를 얻으세요?" (자동 수집→승인 파이프라인 설계 근거)
- **Q F2**: "빵이 유독 많이/적게 나가는 **특별한 날**이 있나요? (밸런타인·화이트데이·어린이날·설·추석)"
- **Q F3**: "**설·추석**엔 매장 문 여나요? 그날 빵 수요는 어떤가요?"

- **Q F4**: "이런 날 **생산량을 미리 조정**하시나요? 어떻게 아세요(작년 기억/본사 지침)?"

## 8. 테마 G — 예약·대량주문 (8분)

> **왜**: 예약이 POS에 안 잡혀(F6) 실수요를 오염시킴. 규모·빈도·기록 위치를 알아야 데이터에서 걷어냄.

- **Q G1**: "**단체 예약·케이터링·대량 주문**이 얼마나 자주 들어오나요? 보통 몇 개 규모?"
- **Q G2 ★**: "예약분은 **판매·발주·폐기 중 어디에** 남나요? 예약 받으면 그만큼 **추가로 발주/생산**하나요, 아니면 원래 발주 안에서 빼주나요?"
  - ↳ 대조: 실무자 아이디어 ②(판매+폐기 > 발주 = 추가발주=bulk). 예약이 발주량에 포함되는지 현장 확인 → 우리 bulk 검출 검증의 핵심.
- **Q G3**: "케이크 **예약**은 빵이랑 완전히 다른 경로인가요?"
  - ↳ 대조: F6 케이크 예약코드 0건.
- **Q G4**: "어떤 빵이 **하루에 평소보다 확 많이** 팔리면 대부분 **예약/단체** 때문인가요, 아니면 그냥 잘 팔린 날도 있나요?"
  - ↳ 대조: 실무자 아이디어 ①(30%↑ 튀면 평균대체). 이 규칙이 진짜 예약만 걸러내는지, 정상 성수기를 오검출하는지 판별.

## 부록 — 진행 팁

- **순서**: A(프로세스)→B(폐기)→C(할인)→D(품절)은 자연스레 이어짐(생산→남음→할인→품절). 이 흐름 유지.
- **리딩 금지**: F1~F9 우리 숫자를 먼저 던지면 점장이 맞춰줌 → 오염. 반드시 **열린 질문 먼저**, 답 들은 뒤 "저희 데이터로는 이렇게 보이는데 맞나요?"로 대조.
- **시간 부족 시 우선순위**: A1·A4(프로세스·상한) → B2(과잉생산 이유) → C2(α 방향) → D2·D3(대체·리필) → F1(xmas 상한) → H1(원가율). 나머지는 후속.
- **꼭 받아올 것**: (1) 현행 발주 결정 방식 1문장 요약 (2) α 방향 직관 (3) 원가율 가용 여부 (4) 예약 기록 위치 (5) 냉동생지 정체.
- **기록**: 답변을 F1~F9 대조표에 바로 매핑하면 후처리 빠름.
