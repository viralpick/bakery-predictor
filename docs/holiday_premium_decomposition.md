# 공휴일 프리미엄 분해 + 헤드라인 feature 결함 발견 (광교 2021-25)

트랙2(명절 prior) 진행 중 사용자(architect)가 "크리스마스=케이크 생산고정 / 다른 특수일=진짜 수요 /
대체공휴일·연휴여행 고려" 우려 제기 → EventLevelPrior 일반화 안전성을 데이터로 검증한 결과,
**개별 이벤트 prior가 아니라 헤드라인 공휴일 feature 파이프라인 수리가 진짜 fix**로 재규정됨.

## ★결정적 발견: 헤드라인 모델이 2021-23 공휴일에 blind
`category_aggregate.add_holiday_features`가 `data/external/calendar_raw.parquet`에서 공휴일을 읽는데,
이 파일이 **2024-2026만 커버 — 2021·2022·2023 공휴일 전부 누락**(`is_public_holiday`가 조용히 0).

| 연도 | raw parquet | holidays.KR 정답 |
|------|------|------|
| 2021 | **0** | 18 |
| 2022 | **0** | 19 |
| 2023 | **0** | 18 |
| 2024 | 19 | 19 |
| 2025 | 19 | 19 |

730d 롤링 백테스트에서 2023~초2024 공휴일 예측 시 학습창(2021-23)의 공휴일 프리미엄을 학습 불가
→ base가 공휴일에 −18.5% 과소예측한 **진짜 원인**. 이벤트 prior가 backtest서 "먹힌" 것은 상당부분
이 blind spot 보상. (이상치 분석 땐 진단 스크립트만 build_calendar_daily로 교체했고 **모델 파이프라인은
깨진 소스 유지** — 이 단절이 드러남.)

## 실수요 구조 분해 (정확한 calendar로 계산 — lift = actual / 로컬 동일요일 baseline)

### 로버스트 신호 (feature로 학습 가능)
- **평일 공휴일 프리미엄 = +35%** (고립 평일 단일공휴일 median lift 1.35 [1.23, 1.43], n=20).
- **주말 공휴일 ≈ 0.95** (프리미엄 없음, 평상주말과 거의 같거나 살짝 아래). 오피스 상권이라 주말 baseline이
  이미 낮음 → 공휴일이 더 못 올림. **연휴 생성 여부와 무관**(연휴형 0.93 ≈ 고립형 0.95).
- **명절 당일 high · 다음날 <1** (추석다음날 0.97·설날다음날 0.81·둘째날 0.74) → `off_position_in_streak`.

### 이벤트 고유성 없음 (핵심)
어린이날 평일 lift 1.24 ≈ **전체 평일공휴일 median 1.25**. 어린이날은 "그냥 평일 공휴일" — 고유 효과
아님. 상위 tier만 예외: xmas 1.52-1.63(공급고정), 신정 1.42, 삼일절 1.38, 추석 1.36.

### 노이즈 신호 (미채택)
- **연휴 길이 → 여행 저조**: 식별 안 됨. 대체월요일 lift 1.07~1.56(median 1.18)로 산발 — 2024 어린이날
  월대체(1.10)은 낮은편이나 유일 아님(광복절2021 1.07, 부처님2023 1.08), 삼일절2025는 1.56로 강함.
  연휴 블록 궤적도 감소·증가 양방향. → `off_streak_length`(여행 축) **미채택**, 문서화만.

## 2024 어린이날 overshoot 정체
여행 아니라 **"평일 앵커(median 317)를 주말 어린이날(≈0.95 수준)에 적용"**. 주말공휴일 프리미엄 없음을
절대-median이 뭉갠 결과. → 개별 이벤트 median 앵커의 구조적 한계.

## 재규정된 방향: feature 파이프라인 수리
학습 가능한 진짜 구조 = **공휴일 × 평일/주말**(+명절 위치). 개별 이벤트 prior(트랙2 원안)는 잘못된 도구
(어린이날 고유성 없음 + 이벤트당 4샘플 vs 공휴일 풀 ~71샘플).
- ✅ `calendar_raw.parquet` → `build_calendar_daily`(holidays.KR, 완전) 교체
- ✅ `is_public_holiday × is_weekend` 상호작용 + `off_position_in_streak`
- ❌ `off_streak_length`(여행) — 노이즈
- **xmas 별도 유지** — 공급 고정(주말 거동이 수요 이벤트와 정반대: 0.87 vs sign-flip). feature로 못 접음.
- **어린이날 prior**(PR 미정, 커밋됨)는 구조적 대체가 측정으로 이길 때까지 유지(2026-28 평일이라 안전).

## 수리 후 재측정 (광교 730d/52folds, 2026-07-18)

### per-event base bias (feature 수리 전 → 후) — base가 공휴일 프리미엄 잡는가
| 이벤트 | 수리 전 base | 수리 후 base | 수리 후 blend |
|--------|------|------|------|
| 어린이날 | WAPE 0.210, bias −0.185 | **0.125, −0.055** | 0.122 |
| 추석 | 0.213, −0.125 | **0.146, −0.046** | 0.120 |
| xmas | 0.210, −0.085 | **0.153, −0.049** | 0.081 |

- **어린이날**: 수리된 base가 거의 해결(bias −0.185→−0.055). prior가 더하는 것 없음(0.125→0.122) = **prior 중복**.
  2024(일) 과대도 base +17% < blend +32% → prior 드롭이 주말 overshoot도 완화.
- **xmas**: prior가 여전히 크게 기여(0.153→0.081) = **공급 고정이라 base가 모르는 부분** → 유지 정당.
- **추석**: base 개선되나 prior 아직 약간 기여(0.146→0.120) = 경계.

### 헤드라인 aggregate (수리된 feature)
| config | WAPE | 매진 | 폐기 |
|--------|------|------|------|
| no-prior (base만) | 7.948% | 22.3% | 8.65% |
| xmas만 | 7.890% | 22.0% | 8.72% |
| xmas+childrens+chuseok | 7.799% | 21.7% | 8.79% |
| *(수리 전 broken 현행)* | *7.901%* | *22.5%* | *8.66%* |

- Feature 수리 = 올바른 방향(현행 WAPE 7.901→7.799, 매진 22.5→21.7). **단 aggregate 이동은 작음**(공휴일 연 ~18일 → 365일 총량 희석). 진짜 이득 = per-event 정확도 + 잠재 데이터버그 제거.
- prior는 수리 후에도 WAPE 약간↓지만 WPE/폐기↑(0.61→0.81, surplus 8.65→8.79). 폐기(1차 KPI) 기준 prior는 한계효용 낮음.

### 결론
- ✅ **feature 수리 채택**(데이터버그 fix, 저위험, 올바른 방향). 이 산출물의 본질 = **3년치 공휴일 blind 잠재버그 제거**, "prior 추가"가 아님. aggregate WAPE는 거의 안 움직임(공휴일 ~18일/365)이나 per-event 정확도 + 홀리데이 많은 전향 구간 대비가 실이득.
- **어린이날 prior 유지**: pooled 0.125≈0.122는 **중복 아님** — 주말 fold(2024, prior 해로움 +17%→+32%)와 평일 fold(prior 도움)를 섞은 tie. **평일-only(2026-28 전부 평일) 재계산: base 0.118 → blend 0.088 (~25%↓)**. feature 수리와 prior는 both/and(730d=이벤트 2샘플, 트리가 레벨 못 뽑음 = 원래 EventLevelPrior 정당성). 수리가 겉보기 가치를 부풀렸을 뿐, 과소표본 이벤트엔 여전히 기여.
- **xmas prior 유지**(공급 고정, base 불가능 영역 0.153→0.081).
- 추석 prior 유지(0.146→0.120).
- 회귀: leakage/category 26 통과. 전체 스위트 재실행 필요(add_holiday_features 코어 변경). off_position_in_streak 추가.
