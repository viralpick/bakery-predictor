# 진짜 PoC 범위 정의 — 260605 아티제 실무자 미팅 정리

> 내부 작업 기준 문서. 2026-06-05 아티제 실무자 미팅에서 합의·논의한 내용 + 향후 방향 정리.
> 지금까지(v0~v5)는 **PoC 착수 전 데이터 검증/검토 단계**였고, 이 문서부터가 **실제 PoC의 범위**다.
> 회의 raw 항목 22개를 8개 섹션으로 재구성. 각 항목 번호는 `(N)`으로 추적.

---

## §0. PoC 위상 전환

| 구분 | 지금까지 (v0~v5) | 진짜 PoC (이 문서부터) |
|---|---|---|
| 성격 | 데이터 검증/검토, 가설 탐색 | **실측 검증 + 운영 비교** |
| 산출물 | 분석 리포트, framework 가설 | 광교 실제 발주 권장 + 기존 시스템과 성능 비교 |
| 예측 형태 | 구간(범위) 예측 실험(v5 conformal) | **점추정 + 품절/매진 위험 수치** (§2) |
| 검증 대상 | 4매장 EDA / 다매장 통합 실험 | **광교 단독 성능 검증** (§1) |

핵심 전환: "분석으로 인사이트를 얻는 단계" → "그 인사이트로 **실제 발주에 개입**하고, 기존 운영 대비 나아지는지 4주간 실측하는 단계".

## §1. PoC 운영·검증 설계 — (2)(3)(20)

### 1.1 4주 + 4주 구조 (2)
- **1~4주차 (구축)**: 모델·데이터 파이프라인 정비, 발주 권장량 산출 체계 완성.
- **5~8주차 (검증)**: 우리가 전달한 데이터로 **실제 생산을 진행**. 기존 발주 요청 시스템과 **나란히 성능 비교**.
- 즉 단순 backtest가 아니라 **prospective(전향적) 실측 검증**. 과거 데이터 백테스트의 한계(16 Thursdays noise 등)를 실제 운영으로 보완.

### 1.2 검증 대상 = 광교 단독 (20)
- **PoC 성능 검증의 대상은 광교 1개 매장만.**
- 다른 매장(삼성타운/메세나/광화문 + 신규 유사상권 매장)은 **광교 예측을 돕는 보조 데이터**로만 사용.
- 타매장 예측을 산출할 수는 있으나 **검증 KPI 대상이 아님**.
- → 다매장 통합 학습이 광교 정확도를 높이는지가 보조 데이터 활용의 판단 기준.

### 1.3 비교 baseline = 기존 발주 요청 시스템 (2)
- 우리 모델 권장량 vs 매장/본사의 기존 발주 방식을 **같은 기간·같은 매장**에서 비교.
- "우리가 더 낫다"의 기준은 §1.4 metric으로 정량화.

### 1.4 성능 metric 재협의 (3)
기존 WAPE 중심에서 **운영 KPI 중심**으로 재협의. 후보:
- **폐기 비용 축소** (waste cost ↓)
- **매진 시각 중앙값 증가** (stockout time median ↑ — 늦게 매진될수록 기회손실 적음)
- **매진률 감소** (stockout rate ↓)
- ※ 정확도(WAPE/MASE)는 보조 지표로 유지. 최종 합의는 아티제와 함께(§3).

## §2. 모델 산출물 정의 — (1)(6)(7)(21)

### 2.1 점추정 + 위험 수치 (1) ★ 방향 전환
- 수요 예측은 **범위(구간)가 아니라 단일 수치**로 제공.
- 다만 해당 일자에 대해 **품절 위험·매진 위험을 수치와 함께** 제공.
- → **v5 conformal 구간예측(대칭/비대칭 interval)은 deprecated** (§8 참조). 구간 폭 대신 "이 수량 발주 시 매진 확률 X%, 품절 위험 Y" 형태로 전달.

### 2.2 Bulk(묶음) 예측 → 품목별 비율 (6)
- **단, 수요 이전(흡수)이 통계적으로 검증된 경우에 한해**(§3.5, §4 조건부).
- 수요 이전이 진실이면: 카테고리/묶음 단위로 총량을 먼저 예측 → 그 안에서 **품목별 생산 비율** 결정.
- 수요 이전이 약하면 이 구조는 재검토.

### 2.3 카테고리 정의 = 고객사 제공 (7)
- 카테고리별 수요 흡수를 보려면 **카테고리 경계가 명확해야** 함.
- 우리가 임의로 묶지 않고 **아티제가 카테고리 분류를 결정·제공**.

### 2.4 공통 backbone + 매장별 커스터마이징 (21)
- 매장 특성이 다르므로(과거 EDA에서 확인) **모델 backbone은 동일**하게 가져가되,
- 매장별로 **feature 사용/모델링을 다르게** 할 수 있는 구조가 바람직.
- ※ 단, 과거 검증에서 매장별 과도한 (window, N) customization은 **오버피팅**으로 진단됨 → "구조적 커스터마이징 여지는 두되, 데이터 기반 튜닝은 robust하게" (§8).

## §3. 고객사와 정렬할 기준 — (4)(5)(9)(11)

분석으로 얻은 가정·수치를 일방 확정하지 않고 **아티제와 긴밀히 소통하며 기준을 맞춰나간다 (4)**. 정렬 대상:

| # | 항목 | 현재 우리 추정 | 정렬 필요 |
|---|---|---|---|
| 1 | 마진율 | 미반영(폐기=매진 cost 동일 가정) | 매장/품목별 실제 마진 |
| 2 | 매진·폐기 리스크 및 비용 | 단가 5,000원 고정 가정 | 실제 단가·간접손실(브랜드) |
| 3 | 마감 할인의 실수요 비율 | 광교 20~21시 92% 마감할인 추정 | 실제 회수/실수요 구분 기준 |
| 4 | 수요 전이(한 제품 매진 시 다른 제품으로 전부 전이되는가) | 광교 카테고리 내 거의 흡수 추정 | **통계 검증 후 합의 (§3.5)** |

### 3.5 수요 흡수·이전 통계 검증 (5)
- 수요 이전 여부를 **통계적으로 검증**하고, 그 **결과를 토대로 아티제와 합의**.
- 이 검증 결과가 §2.2(bulk 예측 구조)의 채택 여부를 결정하는 **전제 조건**.

### 3.6 운영·고객행동 인사이트 수집 (9)
- 점주·본사 직원을 통해 **현장 운영 노하우와 고객 행동**을 얻는다.
- 핵심 도메인 사실: **아티제는 카페 성격이 강함 — 빵 사러 오기보다 카페 이용 고객이 다수, 빵은 서브** (§6).

### 3.7 노이즈 제거 기준 현실화 (11)
- 예약 데이터: 지금은 **우리 유추(max≥5 ×2.5 / total≥15)**로 제거 중 → **잠정**.
- 실제 예약 데이터가 없으므로, 본사/매장 협의로 **더 현실에 가까운 유추 기준**을 세운다.
- → 현재 bulk detector 기준은 deprecated 예정(§8), 현실 기준 확보 시 교체.

## §5. 새 feature·모델링 방향 — (10)(12)(13)(16)(17)(18)(19)

### 5.1 고객 수 proxy = 음료 수 (10)
- 아티제는 카페 고객이 다수 → **음료 판매량으로 실제 매장 고객 수를 유추**.
- 수요를 "절대 판매량"이 아니라 "고객 수 대비 빵 판매량"으로 정규화할 여지.

### 5.2 고객 수 대비 빵 판매량 (12) — COO 제안
- 검증 시 **고객 수(음료 수) 대비 빵 판매량 증가**를 주요 지표로 볼 수 있음 (아티제 COO 제안).
- 우리 개입으로 객단가/빵 attach rate가 오르는지 측정 가능.

### 5.3 ⚠️ 단, 이 proxy의 한계 (13) — open question
- 빵 구매는 **다변수**: 배고픔 / 매장 내 대체품 유무 / 시간대 / 간식 vs 주식 등.
- → 음료 수 대비 빵 판매량 지표가 **작동 안 할 가능성** 있음.
- 간접 지표로만(고객 만족도 증가 방향성 정도) 활용. **확정 구현 X, 검토 항목.**

### 5.4 생산량 다중 window (16)
- 하루 생산량을 여러 단위로 본다: **칼로리 / 판매량(개수) / 무게 / 가격(매출)**.
- 단순 개수 외에 무게·칼로리 기준이 매장 capacity·폐기 비용과 더 맞을 수 있음.

### 5.5 운영 노하우·암묵지 피쳐화 (17)(18)
- 점주·본사의 **사전 지식·경험**을 feature로 만들 수 있는 것은 만든다 (17).
- 해당 feature **적용 시 vs 미적용 시 매장 운영 차이를 ablation으로 확인** (18).
- → 이 PoC의 핵심 검증 질문 중 하나: **암묵지 기반 feature가 실제로 얼마나 도움이 되는가** (§7과 연결).

### 5.6 제품 특성 반영 (19) — open question
- 제품을 **식사 대용 / 간식 / 후식** 등 특성으로 구분해 모델에 반영할 수도.
- 단, **경계가 명확하지 않아 애매** — 어떤 비중(soft membership 등)으로 들어갈지 검토 필요. **확정 X, 검토 항목.**

## §6. 도메인 인사이트 (오늘 회의) — (9)(10)(13)

1. **아티제 = 카페 성격이 강함.** 빵 사러 오는 사람보다 카페 이용 고객이 다수, **빵은 서브 아이템.** (9)
2. **고객 수 ≈ 음료 수.** 매장 실제 고객 규모를 음료 판매량으로 유추. (10)
3. **빵 구매는 다변수 위에 있음** — 배고픔/대체품/시간대/간식·주식 여부. 단일 proxy로 설명하기 어려움. (13)

→ 이 도메인 이해가 §5의 feature 설계와 §1.4 metric 선택의 배경.

## §7. 장기 비전 — (22)

- 장기적으로는 **온톨로지 + LLM 기반**:
  - feature 자동 편입/제거, feature 자동 선택
  - 발주량 reasoning
  - → **매장 발주 관련 AI 비서** 형태.
- **이번 PoC의 역할**: 이런 **노하우·암묵지 기반 feature가 얼마나 도움이 되는지 판단** (§5.5와 직결).
- 즉 이번 PoC는 LLM/온톨로지 구현이 아니라, 그 방향이 유효한지 검증하는 **선행 단계**.
